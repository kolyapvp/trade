# Deployment

Для торгового бота здесь используется не blue-green, а `drain deployment`.

Почему:

- У бота есть состояние открытых позиций и внешние побочные эффекты на биржах.
- Параллельный запуск двух активных версий повышает риск дублей, гонок и повторного открытия сделок.
- Текущий бот уже умеет восстанавливать открытые позиции из Redis и Postgres после рестарта, поэтому безопаснее держать один активный инстанс.

Сценарий деплоя:

1. `ops/deploy.sh` обновляет код до целевого коммита.
2. Скрипт выставляет в Redis `tradebot:deployment.status=draining`.
3. Работающий бот перестаёт открывать новые сделки, но продолжает сопровождать и закрывать уже открытые позиции.
4. Скрипт ждёт, пока `tradebot:open_positions` не станет пустым два опроса подряд.
5. После этого пересоздаются `trade-bot`, `prometheus`, `grafana`.
6. После успешного healthcheck статус возвращается в `active`.

Fallback при недоступности PyPI:

- Базовый путь не меняется: `trade-bot` по умолчанию пересобирается на хосте через `docker compose up --build`.
- Если сборка падает на сетевых ошибках к `PyPI`, `ops/deploy.sh` может переключиться на заранее опубликованный образ.
- Для этого нужно передать `FALLBACK_IMAGE_REPOSITORY` и при необходимости registry credentials.
- В GitHub Actions fallback-образ публикуется в `GHCR` с тегом коммита, после чего remote deploy использует его только при сетевом сбое сборки.

Если позиции не закрылись за `DRAIN_TIMEOUT_SECONDS`, деплой завершается ошибкой и статус возвращается в `active`.

Основные переменные:

- `DRAIN_TIMEOUT_SECONDS` — сколько ждать завершения открытых позиций перед отказом деплоя.
- `DRAIN_POLL_SECONDS` — интервал опроса Redis.
- `WAIT_TIMEOUT_SECONDS` — таймаут ожидания `docker compose up --wait`.
- `ROLLING_SERVICES` — список сервисов, которые безопасно пересоздавать в рамках обычного релиза.

Что важно:

- `redis` и `postgres` поднимаются и проверяются, но не включены в обычный rolling restart.
- Если стратегия удерживает позицию очень долго, релиз лучше планировать заранее или повышать `DRAIN_TIMEOUT_SECONDS`.
- При необходимости можно сделать отдельный аварийный сценарий с ручным подтверждением, но он не включён по умолчанию.
