# Deployment

Для торгового бота здесь используется не blue-green, а `drain deployment`.

Почему:

- У бота есть состояние открытых позиций и внешние побочные эффекты на биржах.
- Параллельный запуск двух активных версий повышает риск дублей, гонок и повторного открытия сделок.
- Текущий бот уже умеет восстанавливать открытые позиции из Redis и Postgres после рестарта, поэтому безопаснее держать один активный инстанс.

Сценарий деплоя:

1. GitHub Actions собирает Docker-образ из целевого коммита.
2. Готовый образ публикуется в `GHCR` с тегом коммита и фиксированным digest.
3. `ops/deploy.sh` на хосте обновляет код до целевого коммита.
4. Скрипт выставляет в Redis `tradebot:deployment.status=draining`.
5. Работающий бот перестаёт открывать новые сделки, но продолжает сопровождать и закрывать уже открытые позиции.
6. Скрипт ждёт, пока `tradebot:open_positions` не станет пустым два опроса подряд.
7. После этого хост делает `docker pull` готового образа и пересоздаёт `trade-bot`, `prometheus`, `grafana` без локальной сборки.
8. После успешного healthcheck статус возвращается в `active`.

Основной путь:

- Релизный образ собирается в CI там, где доступ к `PyPI` стабильнее, и публикуется в `GHCR`.
- В production-деплое хост не вызывает `pip install` и не делает `docker build`.
- Для развёртывания используется `DEPLOY_IMAGE`, который передаётся как полный reference вида `ghcr.io/<owner>/trade-bot@sha256:<digest>`.

Резервный путь:

- Если `DEPLOY_IMAGE` не передан, `ops/deploy.sh` сохраняет старое поведение и пытается пересобрать `trade-bot` на хосте через `docker compose up --build`.
- Если такая сборка падает на сетевых ошибках к `PyPI`, `ops/deploy.sh` может переключиться на заранее опубликованный образ через `FALLBACK_IMAGE_REPOSITORY`.
- Для fallback дополнительно можно передать registry credentials.

Если позиции не закрылись за `DRAIN_TIMEOUT_SECONDS`, деплой завершается ошибкой и статус возвращается в `active`.

Основные переменные:

- `DRAIN_TIMEOUT_SECONDS` — сколько ждать завершения открытых позиций перед отказом деплоя.
- `DRAIN_POLL_SECONDS` — интервал опроса Redis.
- `WAIT_TIMEOUT_SECONDS` — таймаут ожидания `docker compose up --wait`.
- `ROLLING_SERVICES` — список сервисов, которые безопасно пересоздавать в рамках обычного релиза.
- `DEPLOY_IMAGE` — полный reference готового образа из registry.
- `DEPLOY_IMAGE_REGISTRY`, `DEPLOY_IMAGE_USERNAME`, `DEPLOY_IMAGE_PASSWORD` — доступ к registry для pull готового образа.

Что важно:

- `redis` и `postgres` поднимаются и проверяются, но не включены в обычный rolling restart.
- Деплой по digest гарантирует, что хост поднимет ровно тот образ, который собрал CI.
- Если стратегия удерживает позицию очень долго, релиз лучше планировать заранее или повышать `DRAIN_TIMEOUT_SECONDS`.
- При необходимости можно сделать отдельный аварийный сценарий с ручным подтверждением, но он не включён по умолчанию.
